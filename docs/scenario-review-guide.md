# シナリオ見直しガイドライン

## 目的

`/eval` の固定シナリオへの過適合を防止するため、定期的にシナリオの妥当性を確認し、必要に応じて新シナリオを追加する。

## 見直しの観点（3 項目）

### 1. 新ユースケースの反映
- 最後のシナリオ更新以降に、新しいカテゴリ（task_type）やワークフローパターンが追加されたか
- 追加されたなら、そのパターンをカバーするシナリオが必要
- 例: `network_design` カテゴリが追加された → T4_network シナリオを検討

### 2. 評価基準の現在性
- `eval/rubric.json` の評価軸が現在の品質目標と一致しているか
- 新しい品質基準（例: Troubleshooting セクションの有無）が追加された場合、チェックリストに反映すべき
- `eval/SCORING_GUIDE.md` のアンカーポイントが実態と乖離していないか

### 3. 多様性の確保
- 現在のシナリオが特定のパターンに偏っていないか
- T1（research）、T2（implement）、T3（proposal）で 3 カテゴリをカバーしているが、以下の観点も検討:
  - フェーズ数の多寡（1-2 フェーズ vs 5+ フェーズ）
  - pre-docs の有無（インテークスキップ vs フル質問）
  - 既存スターターの利用有無

## 見直し手順（step-by-step）

### Step 1: 現状把握
1. `eval/scenarios/` の全シナリオを確認
2. `eval/summary.csv` で過去のスコア推移を確認
3. 最後のシナリオ更新日を確認

### Step 2: 過適合の兆候チェック
以下のいずれかに該当すれば、シナリオの追加・更新を検討:

| 兆候 | 判定基準 |
|------|----------|
| スコア安定 | 直近 3 回の全軸スコア変動が ±0.5 未満 |
| 満点到達 | いずれかの軸で 5.0 が連続 2 回以上 |
| カテゴリ未カバー | init-task SKILL.md がサポートするカテゴリで、シナリオが存在しないものがある |
| ワークフロー未カバー | `/re-init-task`、`/add-feature` 等の主要ワークフローが評価対象になっていない |

### Step 3: 新シナリオの設計（必要時）
1. 既存シナリオと重複しないテーマを選定
2. `eval/scenarios/T{N}_{name}/` ディレクトリを作成
3. 以下のファイルを作成:
   - `task.md`: シナリオの入力（タスク記述）
   - `expected_manifest.json`: 期待される成果物一覧
   - `checklist.json`: 評価チェックリスト

### Step 4: 記録
1. `eval/reports/` に見直しレポートを作成（日付入り）
2. `eval/summary.csv` の `last_scenario_update` を更新
3. CHANGELOG があれば更新

## 既存シナリオ保護ルール

- **既存シナリオ（T1, T2, T3）の削除は禁止**
- 既存シナリオの `task.md` 変更は禁止（スコアの時系列比較が不可能になる）
- 既存シナリオの `checklist.json` への項目追加は許可（削除は禁止）
- 新シナリオは T4 以降の番号で追加

## 過適合の定義と対処

### 過適合とは
評価スコアは高いが、実際のプロジェクト（private_learning_website 等）での運用品質が向上していない状態。

### 対処法
1. 固定シナリオのスコアだけでなく、実プロジェクトの `/retrospective` 結果と比較する
2. 実プロジェクトで発生した問題が、固定シナリオのチェックリストで検出可能かを確認
3. 検出不可能なら、その問題パターンを新シナリオまたは既存チェックリストの追加項目として反映
