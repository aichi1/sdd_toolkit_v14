# Constitution — sdd_toolkit_v14

このドキュメントには二つの役割がある。

1. **ビルド時の憲法**: v14 を実装する間、Builder/Validator/専門エージェントが従う非交渉の原則。
2. **成果物としての憲法の雛形**: v14 は Constitutional SDD を機能として持つ。
   本書はその `constitution.md`(下流プロジェクトに配布される既定の憲法)の初期版でもある。

各条項は「原則」と「強制点(どのノード/フック/評価で機械的に検査するか)」をセットで持つ。
強制点を持たない原則は憲法に載せない。

---

## 第1条: 仕様が唯一の真実源

- **原則**: 実装は仕様(`docs/spec.md`)から導出される。コードと仕様が食い違う場合、直すのはコードであって仕様ではない。
- **強制点**: `verify` ノードが各成果物を仕様の受入基準に照合する。Validator は仕様条項を引用しない指摘を出してはならない。

## 第2条: 状態は薄く保つ

- **原則**: LangGraph の状態にはパスと ID のみを載せ、本文(diff・生成コード・トレース)はディスク(worktree)に置く。1 チェックポイントは 10KB 未満を目標とする。
- **理由**: SqliteSaver はチェックポイントごとに状態全体をシリアライズする。肥大は書込みレイテンシを 15ms から数百 ms に悪化させ、エージェント自体のボトルネックになる。
- **強制点**: `harness/` のチェックポイントサイズ計測。状態スキーマのレビュー(`03_plan.md` の `TaskState` を逸脱する本文フィールドの追加を禁止)。

## 第3条: 副作用は承認の後にしか実行しない

- **原則**: main への merge、外部 API 呼び出し、課金を伴う操作などの不可逆な副作用は、`review` ノードの `interrupt()` を通過した後の分岐にのみ書く。`interrupt()` より前は冪等な計算だけにする。
- **理由**: resume 時にノードは頭から再実行される。副作用が interrupt の前にあると二重実行される。
- **強制点**: `review` ノードのコードレビュー。副作用関数(`merge_worktree` 等)が承認分岐の内側にあることを Validator が確認する。

## 第4条: すべてのコード実行は隔離する

- **原則**: エージェントの作業は git worktree で物理分離し、テスト/コード実行は podman rootless の使い捨てコンテナ(読み取り専用マウント、egress 遮断)で行う。
- **強制点**: `sandbox.py` の実装レビュー。ネットワーク egress が既定で遮断されていることの確認。ホストで直接コードを実行するパスを持たないこと。

## 第5条: 守りは二層で敷く

- **原則**: ソフト境界(Claude Agent SDK の hooks)とハード境界(podman/cgroups)を両方持つ。片方だけに依存しない。
- **強制点**: `PreToolUse` フックが破壊的操作(`rm -rf`, `git push`, `.env` 読取り等)を拒否すること、かつ cgroups v2 で各実行体に `MemoryMax` が割り当てられていることを、両方確認する。

## 第6条: 既存資産を再利用し、二重化しない

- **原則**: v12/v13 の `handlers.py`(KB ロジック層)、FastMCP の実装様式、Builder/Validator の定義、`eval/` を再利用する。同じ責務を REST と MCP で二重に持つような重複を作らない。
- **理由**: v13 で「MCP が名ばかりで REST と二重化していた」問題を修正した経緯を繰り返さない。
- **強制点**: **既存資産と同一の責務/コーパスを扱う** MCP サーバは、その責務を再実装せず `handlers.py` 等のロジックを import して薄いファサードにすること。同一ロジックの再実装（二重化）を Validator が検出したら FAIL。
  - **適用範囲の明確化（2026-07-01 改正 / Phase 3）**: この強制点は「既存資産が既にカバーする責務」に対して働く。`handlers.py` は **BM25 × KB(`~/.sdd-knowledge`)** 専用であり、`context_server` の **ChromaDB × 現プロジェクト仕様(`docs/*.md`)スライシング**は**別コーパス・別関心事＝新規責務**なので、handlers.py の import は不要であり、handlers を import しないことは二重化ではない（=第6条違反ではない）。判定基準は「import したか」ではなく「**既存責務を再実装して二重化したか**」。新規 MCP サーバも薄いファサードに徹し、独自のビジネスロジック肥大を避けること。

## 第7条: 評価でリリースをゲートする

- **原則**: プロンプトやモデルの変更を「雰囲気」で出さない。`eval_suite` のスコアと回帰検出を通過してから次フェーズへ進む。
- **強制点**: `eval` ノードが閾値未満なら `build` へ戻す条件エッジ。フェーズ完了時の `/eval` による v13 ベースライン比較。

## 第8条: 可観測性は必須

- **原則**: すべての run はトレースされ、コストとトークン(サブエージェントはモデル別内訳)が記録される。
- **強制点**: 各 run が LangSmith または OTel にトレースを出していること。`ResultMessage` の `total_cost_usd` が保存されていること。

## 第9条: 外殻は決定論的、内側は確率的

- **原則**: ジョブのライフサイクル・分岐・リトライ・承認ゲートは LangGraph(決定論的)で制御する。モデルにループを駆動させる部分は Claude Agent SDK(確率的)に閉じる。両者を混同しない。
- **強制点**: 状態遷移がグラフのエッジとして明示されていること。エージェント内部の非決定性が状態機械の制御フローに漏れていないことのレビュー。

## 第10条: セキュリティは構築時から

- **原則**: 生成物のセキュリティは後付けにしない。security 専門エージェントと `eval_suite` が CWE/OWASP マッピングに沿って走査する。
- **強制点**: `verify` ノードに security サブエージェントが含まれること。既知脆弱性クラスの検査項目が `eval_suite` に存在すること。

---

## 改正手続き

憲法の変更は仕様変更として扱う。`docs/spec.md` の更新 → `/run-phase` での再検証を経ずに条項を追加・削除してはならない。
下流に配布される `constitution.md` は、本書からビルド専用の記述(第6条の v12 経緯など)を除いたサブセットとして生成する。
